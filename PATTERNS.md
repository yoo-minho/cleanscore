# 결함 패턴 카탈로그

cleanscore가 잡는 패턴들의 정본. **점수는 흐릿한 집계, 이 카탈로그는 정확한 목록이다.**

각 패턴은 "외부 기여(PR)로 안전한가"로 등급이 갈린다. 남의 코드베이스에 drive-by PR을
넣어도 되는 패턴과, 코드 주인만 판단할 수 있는 패턴은 다르다.

---

## PR 안전 조건 (4가지 전부 충족해야 T1/T2)

| # | 조건 | 왜 |
|---|------|-----|
| 1 | **정답이 하나** | 취향이 개입하면 리뷰가 논쟁이 된다. "리팩터했어요"는 거절당한다 |
| 2 | **기계적 변환** | 아키텍처 결정이 필요하면 주인의 몫이다 |
| 3 | **검증 가능** | 고친 뒤 같은 도구로 사라지는지 확인할 수 있어야 한다 |
| 4 | **n이 큰 자리** | ← 실전에서 얻은 조건. 정적 분석은 n을 모른다 |

**4번이 핵심이다.** 패턴을 찾는 것과 그 패턴이 *중요한지*는 다르다.
엔티티 컬렉션(레코드 수만큼 커짐) 위의 O(n²)는 진짜 문제고,
설정·필드 목록(보통 5개) 위의 O(n²)는 "성급한 최적화"라는 소리를 듣는다.

## 티어

| 티어 | 정의 | PR |
|------|------|-----|
| **T1** | n과 무관하게 **틀린 코드**(버그) | ✅ 항상 |
| **T2** | 성능 결함이고 **n이 컬렉션 크기만큼** 커짐 | ✅ 검증 후 |
| **T3** | 패턴은 맞지만 **n이 유한**(설정·필드 목록) | ❌ 보고만 |
| **OWNER** | 맥락·취향이 필요 — 코드 주인만 판단 가능 | ❌ 절대 |

## 카탈로그

### T1 — 버그

#### `await` in `.forEach()`
- **탐지**: `.forEach(cb)`의 콜백 본문(중첩 함수 제외)에 `await`가 있음
- **왜 버그**: `forEach`는 콜백의 프라미스를 무시한다. 저자는 기다린다고 믿지만 안 기다린다
- **고침**: `for...of` (순차 필요 시) 또는 `Promise.all(map(...))` (병렬 가능 시)
- **오탐 가드**: 중첩 함수 안의 `await`는 그 함수의 문제이므로 제외
- **실적**: twenty·cal.com 실측 0곳 (잘 관리된 repo엔 희귀 — 린터가 잡아줌)

#### 전역 플래그 정규식 + 루프 안 `.test()`
- **탐지**: `/g`(또는 `/y`) 플래그 정규식을 **루프 밖에 선언**해두고, 루프·콜백 안에서 `.test()`
- **왜 버그**: `/g` 정규식의 `.test()`는 `lastIndex`를 전진시킨다. 다음 호출은 **다른 문자열인데도 그 위치부터** 찾는다 → 같은 입력도 호출마다 결과가 뒤바뀐다. 성능이 아니라 **조용히 틀린 답**
- **고침**: `/g` 제거, 또는 매 회 새 정규식, 또는 `.test()` 전에 `lastIndex = 0`
- **오탐 가드**: **`.exec()`는 제외한다.** `while ((m = re.exec(s)) !== null)` 은 `/g`의 *정석 순회 관용구*다 (twenty에서 실제로 `lastIndex = 0` 리셋까지 하고 있었다)

#### 배열에 `for...in`
- **탐지**: 배열이라는 증거(리터럴·`map`·`filter`·`split` 등)가 있는 변수에 `for...in`
- **왜 버그**: 인덱스가 **문자열**이고, 상속 속성까지 돌며, 순서 보장이 없다
- **고침**: `for...of` / `forEach`
- **오탐 가드**: 객체에 쓰는 `for...in`은 정상 — **배열이라는 증거가 있을 때만** 잡는다

#### ~~floating promise~~ → **티어 정정: 주인 의도 필요**
- **탐지**: async 함수 안에서, 로컬 `async` 함수를 `await` 없이 호출하고 결과를 버림(ExpressionStatement)
- **왜 T1이 아닌가**: 처음엔 "`await` 한 단어면 끝"이라 T1로 판정했다. 실물을 보니
  **fire-and-forget이 의도인 경우가 있고**, 고치는 법이 `await` / `.catch()` / `void` 셋 중 선택 —
  응답 지연 vs 신뢰성의 **설계 결정**이다. 기계적이지 않다
- **실측**: cal.com 2곳(`sendCancellationEmail`·`scheduleTrigger` — 서버리스면 유실 위험이 실재),
  formbricks 2곳(React 이벤트 핸들러 = 거의 확실히 의도적)
- **처리**: 진단으로만 보고. PR은 주인의 의도를 알 때만

### T2 — 성능, n 무한

#### 루프 안 DB/HTTP 순차 `await` (N+1)
- **탐지**: 루프 안에서 데이터 호출(`findMany`·`findOne`·`query`·`execute`·`fetch` 등)을 **직접 await**
- **고침**: 루프 앞에서 `IN`/`$in`으로 한 번에 조회 → Map으로 조회, 또는 배치 쓰기
- **오탐 가드**:
  - `Promise.all(items.map(...))` 안의 개별 호출은 **직접 await가 아니므로 제외** — 병렬 배칭은 올바른 패턴이다
  - **재시도 루프**(`while (!acquired)`, 락 획득·이름 중복 탐색)는 N+1이 아니다 → 사람이 검증
  - 이름만 보고 리더로 판정하지 않는다 (아래 "이름 충돌" 참고)
- **실적**: [immich #30163](https://github.com/immich-app/immich/pull/30163), [novu #12074](https://github.com/novuhq/novu/pull/12074)

#### 루프 안 파일 읽기
- **탐지**: 루프 안에서 `readFileSync`/`readFile`/`readdirSync`, 또는 그것을 2홉 내로 호출하는 함수
- **고침**: 루프 밖에서 한 번 읽어 캐시
- **오탐 가드**: 모듈 캐시(`Map` get/set, 메모 변수)를 거치면 제외

#### O(n²) 배열 조회
- **탐지**: 루프 안에서 **루프 밖에 선언된 배열**에 `find`/`findIndex`/`some`(조회) 또는 `filter`(그룹핑)
- **고침**: 루프 앞에서 1회 인덱싱 → O(1). 조회는 `Map<key, T>`, **`filter`는 `Map<key, T[]>` 그룹핑**
- **오탐 가드**:
  - **배열 전용 메서드만** 본다(`find`/`findIndex`/`some`) → `str.includes` 같은 문자열 오탐 원천 차단
  - 루프 **안에서 만들어진** 배열은 제외(매 회 새로 만들어지므로 인덱싱 불가)
  - ⚠️ **`.find`는 첫 일치, `new Map(arr.map(...))`는 마지막 일치** — 중복 키가 있으면 의미가 달라진다.
    고칠 때 *첫 일치 우선*을 보존할 것
- **실적**: [cal.com #29828](https://github.com/calcom/cal.diy/pull/29828), [twenty #23231](https://github.com/twentyhq/twenty/pull/23231), [twenty #23232](https://github.com/twentyhq/twenty/pull/23232)(GraphQL 관계 하이드레이션 핫패스)

#### 진짜 반복문 안 `new RegExp`
- **탐지**: `for`/`while`/`do` 안의 `new RegExp(...)`
- **고침**: 루프 밖으로 호이스팅
- **오탐 가드**: **`.map()`은 루프로 치지 않는다** — 모듈 로드 시 1회 도는 캐시 빌드(`CACHED_REGEX_PATTERNS = entries.map(... new RegExp ...)`)를 재컴파일로 오인했던 실제 사례가 있다

### T3 — 보고만, PR 금지

#### 스프레드 누적
- **탐지**: 루프/`reduce`에서 `acc = [...acc, x]` 또는 `{...acc, k: v}`
- **왜 T3**: 패턴 자체는 O(n²)가 맞지만, 실측해보면 대부분 **설정·필드 목록**(n≈5) 위에서 일어난다.
  twenty 55곳·cal.com 7곳을 전수로 보니 GraphQL 선택 필드, 스키마 옵션 같은 유한 컬렉션이었다
- **언제 T2로 승격되나**: 누적 대상이 **레코드/엔티티 컬렉션**임이 확인될 때. 사람이 판단

### OWNER — 코드 주인 전용

| 축 | 왜 PR 불가 |
|----|-----------|
| **인지 복잡도** | "이 함수 복잡해서 쪼갰습니다" = 취향. 외부인 drive-by는 거절된다 |
| **중복** | 헬퍼 추출은 성급한 추상화일 수 있다. 구조적 유사와 복붙은 다르다 |
| **데드코드** | knip "미사용"의 대부분은 **공개 API·엔트리·WIP**다. 코드 밖 호출자(크론·웹훅·인쇄된 QR)는 정적 분석이 원리상 못 본다 → `// @keep`으로 주인이 표시 |

## 알려진 오탐 클래스 (실전에서 밟은 지뢰)

| 오탐 | 증상 | 대응 |
|------|------|------|
| **이름 충돌** | 배열의 `.push()`·Map의 `.get()`이, 같은 이름의 유저 함수(리더)로 해석됨 → 루프 IO **834곳** 오검출 | 빌트인 메서드명 제외 + **파일 스코프** 리더 해석 (834→36) |
| **`.map()` = 루프** | 모듈 로드 1회 캐시 빌드를 "매 회 재컴파일"로 오인 | 진짜 반복문에서만 판정 |
| **`Promise.all(map(async))`** | 병렬 배칭인데 순차 N+1로 보임 | 직접 await만 카운트 + 사람 검증 |
| **재시도/탐색 루프** | 락 획득 `while`, 이름 중복 `while`을 N+1로 오인 | 사람 검증 (정적으로는 구분 불가) |
| **가짜 조인** | `for a { for b { if (b.k === a.scalar) } }` 를 조인으로 오인 — 안쪽이 *탐색*이 아니라 이미 필터된 배열 순회였다 | 축 자체를 **폐기**(가드가 어렵고 실적 0). 다만 이 오탐이 진짜 패턴(`filter` 그룹핑)을 찾아줬다 |
| **미니파이·번들 파일** | 저장소에 커밋된 벤더 번들·시드 에셋(`seed-project/index.mjs` 등)이 git 추적 대상이라 모든 필터를 통과 → for...in **129곳** 오탐, 게다가 **점수 자체를 왜곡**(twenty 78 B→86 A, maxCog 356→97, 중복 15.7%→9.5%) | 파일명 관례 + "한 줄이 비정상적으로 김"으로 판정해 전역 제외 |
| **정석 관용구를 버그로** | `while ((m = re.exec(s)) !== null)` 는 `/g` 정규식의 올바른 순회법인데 "루프 안 정규식 호출"로 잡힘 | `.exec()` 제외, `.test()`만 판정 |
| **테스트·벤치 파일** | knip이 `.test-d.ts`·`benchmarks/`를 "미사용"으로 보고 → 데드 **181곳**(파일 수보다 많음) | 비-프로덕션 파일을 모든 축에서 일관 제외 + 분석 대상과 교집합만 집계 |

## 패턴 추가 규칙

새 패턴을 넣을 때 반드시:

1. **티어를 먼저 정한다.** T3면 PR 대상이 아니다 — 보고만 한다
2. **오탐 가드를 같이 쓴다.** 가드 없는 패턴은 넣지 않는다 (가드가 패턴의 절반이다)
3. **합성 테스트로 검증**한다 — 잡아야 할 것을 잡고, 잡으면 안 되는 것을 안 잡는지
4. **실물 repo에서 확인**한다. 0곳이면 "없는 것"인지 "디텍터가 죽은 것"인지 반드시 구분
5. PR을 낸다면 **사람이 손으로 검증**한다. 정적 분석은 후보만 뽑는다

> 실적: 스캔한 저장소 ~24개 중 안전하게 PR 가능한 결함은 **5건**이었다. 오탐 클래스는 **10종** 발견했다.
> 성숙한 프로젝트일수록 깨끗하고, 걸린 것도 대부분 재시도 루프·병렬 배칭·트랜잭션 내부라
> 손댈 수 없었다. **이게 현실 적중률이다.**
