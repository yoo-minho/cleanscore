# 패턴 루프 — 측정을 정직하게 키우는 방법

cleanscore는 "축을 더 넣으면 좋아지는" 도구가 아니다. 축은 넣기 쉽고, **틀린 축도 넣기 쉽다.**
그래서 축을 늘리는 절차 자체에 강제장치를 건다.

> **핵심 명제: "이걸 PR로 낼 수 있나?"가 지표를 정직하게 만든다.**
>
> 점수는 자기가 자기를 채점한다 — 계수를 만지면 원하는 결과가 나온다.
> 머지는 조작할 수 없다. 남의 저장소 메인테이너가 눌러야 한다.
> 그래서 **머지 가능성**을 축의 검증 기준으로 쓴다.
> 실제로 이 프로젝트의 오탐 5종은 전부 유닛 테스트가 아니라 **PR을 내보려다** 발견됐다.

---

## 루프

```
① 패턴 가설 → ② 티어 판정 → ③ 디텍터+가드 구현 → ④ 합성 검증
   → ⑤ 실물 스캔 → ⑥ 손 검증 → ⑦ 오탐이면 가드 추가(⑤로) 
   → ⑧ PR → ⑨ 머지 = 임팩트 +1 → ⑩ 카탈로그 기록 → ①
```

### ① 패턴 가설
코드를 읽다 눈에 걸린 것, 알려진 안티패턴, 다른 축을 만들다 옆에서 본 것.
> 실제: `.find`를 루프에서 쓰는 걸 documenso·novu 코드를 *읽다가* 봤다. 도구가 아니라 눈이 먼저였다.

### ② 티어 판정 — **구현 전에**
[PATTERNS.md](./PATTERNS.md)의 4조건으로 T1/T2/T3/OWNER를 먼저 정한다.
**T3·OWNER면 점수에 넣지 않고 진단으로만** 만들거나, 아예 안 만든다.
> 실제: 스프레드 누적을 T1이라 믿고 만들었는데, 실물을 보니 전부 유한 컬렉션(n≈5) → T3.
> **구현 전에 티어를 정했다면 순서가 달랐다.** 이 실패가 4번째 조건("n이 큰 자리")을 낳았다.

### ③ 디텍터 + 가드 동시 구현
**가드 없는 디텍터는 만들지 않는다.** 가드가 패턴의 절반이다.
> 실제: O(n²) 축은 처음부터 "배열 전용 메서드만"(문자열 오탐 차단), "루프 안에서 만든 배열 제외"를 넣고 시작해서 오탐이 거의 없었다.
> 반대로 io 축은 가드 없이 시작해 **834곳 오탐**을 냈다.

### ④ 합성 검증
잡아야 할 것을 잡고, **잡으면 안 되는 것을 안 잡는지** 둘 다 본다.
> 실제: 병적 코드 대조군(중복 100%)을 만들어 점수가 최하위로 떨어지는지 확인했다.

### ⑤ 실물 스캔
실제 저장소 여러 개에 돌린다. **0곳이 나오면 반드시 둘 중 뭔지 가른다:**
- (a) 그 저장소에 진짜 없다
- (b) 디텍터가 죽었다

> 실제: `await in forEach`가 twenty·cal.com에서 0곳. 합성 테스트로 (a)임을 확인했다.
> 이 단계를 건너뛰면 죽은 디텍터를 "깨끗하다"로 오독한다.

### ⑥ 손 검증 — 생략 금지
**정적 분석은 후보만 뽑는다.** 상위 몇 건을 직접 읽는다.
> 실제 적중률: immich에서 `execute` 4곳 중 **3곳이 오탐**(락 재시도 루프)이었고 1곳만 진짜였다.
> 검증 없이 4건을 PR했다면 도구 신뢰가 끝났다.

### ⑦ 오탐이면 → 가드 추가 → ⑤로
오탐은 실패가 아니라 **가드의 재료**다. 클래스로 일반화해서 카탈로그에 남긴다.
> 실제: "이름 충돌"을 잡으려 빌트인 메서드 제외 + 파일 스코프 해석을 넣었더니 **834 → 36**.
> 이 가드는 io 축뿐 아니라 이름 기반 해석을 하는 모든 축에 적용된다.

### ⑧ PR — T1/T2 + 손 검증 통과분만
- 상대 저장소의 **템플릿·라벨·기여 규칙을 지킨다** (immich는 템플릿 미준수로 봇이 자동 closed 했다)
- **LLM 사용을 정직하게 공개**한다 (요구하는 프로젝트가 있다)
- 검증 못 한 건 **검증 못 했다고 쓴다.** 체크박스를 거짓으로 채우지 않는다
- 외부 PR을 안 받는 프로젝트면(documenso) **깨끗이 포기한다**

### ⑨ 머지 = 임팩트 +1
`node bin/impact.mjs`가 GitHub에서 상태를 읽어 [IMPACT.md](./IMPACT.md)와 랜딩을 갱신한다.
**머지되지 않는 축은 중요하지 않은 축이다.**

### ⑩ 카탈로그 기록
패턴 + 가드 + 밟은 오탐을 [PATTERNS.md](./PATTERNS.md)에 남긴다.
기록하지 않으면 **같은 지뢰를 다시 밟는다.**

## 실패 모드 (전부 실제로 겪음)

| 실패 | 결과 | 방지 |
|------|------|------|
| 손 검증 생략 | 오탐 PR → 도구 신뢰 붕괴 | ⑥ 무조건 |
| "0곳"을 안 가름 | 죽은 디텍터를 "깨끗함"으로 오독 | ⑤ 합성 대조 |
| 티어를 나중에 판정 | T3를 T1로 착각하고 구현 | ② 구현 전에 |
| 가드 없이 출시 | 834곳 오탐 | ③ 동시 구현 |
| 진단 축을 점수에 편입 | 보정이 흔들리고 전 프로젝트 점수가 요동 | 진단은 진단으로 |
| 상태 조회 실패를 그대로 기록 | 레이트리밋 때 `unknown`이 로그를 오염 | 못 읽으면 **안 쓴다** |

## 언제 축을 강등/폐기하나

- 저장소 N개를 스캔해 **PR 가능한 건이 0**이면 → T3로 강등(진단만) 또는 폐기
- 실측상 **거의 발동하지 않으면**(존경받는 OSS 17종에서 0회) → 점수에서 제외
  > 실제: `renderGates` 축은 17개 저장소에서 한 번도 발동하지 않아 점수에서 뺐다. 죽은 축은 공짜 점수를 준다.

## 현실 적중률 (2026-07 기준)

```
스캔한 저장소     ~13개 (라이브러리 14 + 앱/백엔드 10)
PR 가능한 결함     4건
축                6개 (점수 4 + 진단 2군)
발견한 오탐 클래스  5종
```

성숙한 프로젝트일수록 깨끗하다. 걸린 것도 대부분 재시도 루프·병렬 배칭·트랜잭션 내부라
손댈 수 없었다. **이 숫자를 낮게 보이지 않게 포장하지 않는다** — 그게 이 루프의 요점이다.
